# 波动率预测模型 - 实盘部署方案

## 一、项目背景与目标

### 1.1 现状回顾

当前已完成的模块：

| 模块 | 文件 | 状态 | 说明 |
|------|------|------|------|
| 历史特征生成 | `features_label.py` | ✅ 完成 | Polars 向量化批量处理 |
| 实时特征计算 | `RealtimeFeatureComputer` | ✅ 完成 | Numba + 环形缓冲区 |
| 模型训练 | `train.py` | ✅ 完成 | LightGBM + QuantileTransformer |
| 结果保存 | `save_daily_results` | ✅ 完成 | parquet + npz + 模型序列化 |

**核心技术栈**：
- 数据采样频率：20ms
- 波动率窗口：1, 3, 5, 10, 15, 30, 60, 120 分钟
- 模型：LightGBM Regressor
- 目标标签：未来 5 分钟波动率 (`y_vol_5m`)
- 模型更新频率：每日 (UTC 00:00)

### 1.2 实盘部署目标

- Python 服务长期运行，接收 Go 程序推送的 20ms tick 数据
- 实时计算波动率特征并输出预测值
- 与 Go 程序通过 ZeroMQ 进行进程间通信
- 每日自动重训练模型，服务端无感知热更新模型

## 三、通信协议

### 3.1 选择 ZeroMQ 的原因

- 已有成熟代码：`zeromq-server_client_test_01.py`
- 支持进程间高速通信
- PUSH/PULL 模式天然适合生产-消费场景
- 支持 JSON 消息格式（orjson 高性能序列化）

### 3.2 消息格式

**Go → Python (tick 数据)**：

```json
{
  "type": "bookTicker",
  "symbol": "XRPUSDT",
  "bidPrice": "0.51234",
  "askPrice": "0.51245",
  "timestamp": 1704067200000000000
}
```

字段说明：
- `bidPrice`：买一价（字符串）
- `askPrice`：卖一价（字符串）
- `timestamp`：时间戳（纳秒）

**Python → Go (预测结果)**：

```json
{
  "symbol": "XRPUSDT",
  "timestamp": 1704067200000000000,
  "pred_vol": 0.002345,
  "status": "ok"
}
```

字段说明：
- `symbol`：币种
- `timestamp`：对应 tick 的时间戳
- `pred_vol`：波动率预测值（如处于预热期则不返回）
- `status`：状态
  - `"warmup"`：预热期（前 2 小时），特征不足
  - `"ok"`：正常预测

### 3.3 ZeroMQ 配置

```python
# Python 端
context = zmq.Context()
receiver = context.socket(zmq.PULL)   # 接收 Go 的 tick
receiver.bind("tcp://0.0.0.0:5555")

sender = context.socket(zmq.PUSH)    # 发送预测结果给 Go
sender.connect("tcp://127.0.0.1:6666")
```

```go
// Go 端
sender := zmq.NewPushSocket()
sender.Connect("tcp://127.0.0.1:5555")    // 发送 tick

receiver := zmq.NewPullSocket()
receiver.Bind("tcp://0.0.0.0:6666")        // 接收预测结果
```

## 四、核心模块设计

### 4.1 RealtimeFeatureComputer（已有，需确认一致性）

#### 4.1.1 功能说明

使用环形缓冲区和 Numba 实现毫秒级特征计算，维护 120 分钟的滚动窗口。

#### 4.1.2 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大缓冲区 | 360000 ticks | 120min × 60s × 1000ms / 20ms |
| 预热时间 | 7200000000000 ns | 2小时 |
| 增量计算复杂度 | O(1) | 不遍历历史，只更新当前值 |

#### 4.1.3 增量波动率算法

```python
# 伪代码
def compute_rolling_std_incremental(valid_data, window):
    s = sum(valid_data[:window])
    s2 = sum(valid_data[:window] ** 2)
    mean = s / window
    var = s2 / window - mean ** 2
    return sqrt(var) if var > 0 else 0
```

#### 4.1.4 与历史批处理的一致性验证

| 特征 | Polars 批处理 | Numba 实盘 | 一致性 |
|------|---------------|------------|--------|
| rolling_std | `rolling_std(window_size=N)` | 增量公式 | ✅ 需验证 |
| 波动率比率 | 条件判断 | 同逻辑 | ✅ |
| 时间特征 | sin/cos 公式 | 同公式 | ✅ |

### 4.2 模型推理模块

#### 4.2.1 模型加载

```python
import lightgbm as lgb
import joblib
import json

class ModelLoader:
    def __init__(self, model_dir):
        self.model_dir = Path(model_dir)
        self.model = None
        self.qt = None
        self.feature_cols = None
        self.mtime = 0
        self.load()

    def load(self):
        self.model = lgb.Booster(model_file=str(self.model_dir / "model.txt"))
        self.qt = joblib.load(self.model_dir / "quantile_transformer.pkl")
        with open(self.model_dir / "feature_cols.json") as f:
            self.feature_cols = json.load(f)
        self.mtime = (self.model_dir / "model.txt").stat().st_mtime
```

#### 4.2.2 预测推理

```python
def predict(self, features: dict) -> float:
    X = np.array([[features[c] for c in self.feature_cols]])
    y_qt = self.model.predict(X)
    y_pred = self.qt.inverse_transform(y_qt.reshape(1, -1)).flatten()[0]
    return float(y_pred)
```

### 4.3 模型热更新机制

#### 4.3.1 设计思路

- 服务端不重启，通过文件时间戳检测模型更新
- 每次收到 tick 时检查一次 `model.txt` 的 mtime
- mtime 变化则重新加载模型文件
- 加载过程在主线程完成（模型文件小，加载极快）

#### 4.3.2 实现

```python
def check_and_reload(self):
    model_path = self.model_dir / "model.txt"
    current_mtime = model_path.stat().st_mtime
    if current_mtime > self.model_mtime:
        logger.info("检测到新模型，开始 reload...")
        self.model = lgb.Booster(model_file=str(model_path))
        self.qt = joblib.load(self.model_dir / "quantile_transformer.pkl")
        with open(self.model_dir / "feature_cols.json") as f:
            self.feature_cols = json.load(f)
        self.model_mtime = current_mtime
        logger.info("模型 reload 完成，预测服务继续运行")
```

#### 4.3.3 风险控制

- 加载失败时保留旧模型，继续服务
- 记录 reload 次数和成功/失败状态

## 六、预热期处理

### 6.1 预热时长

最大波动率窗口为 120 分钟，因此需要至少 2 小时的历史数据才能开始正常预测。

### 6.2 预热判断逻辑

```python
# 在 RealtimeFeatureComputer 中
MIN_TICKS = 120 * 60 * 1000 // 20  # 360000 ticks

def predict(self, bid, ask, ts_ns):
    self.computer.update(bid, ask, ts_ns)

    if self.computer.buffer_count < MIN_TICKS:
        return {"status": "warmup", "buffer_count": self.computer.buffer_count}

    features = self.computer.compute_features()
    pred_vol = self.model_loader.predict(features)
    return {"status": "ok", "pred_vol": pred_vol}
```

### 6.3 Go 端处理

Go 程序收到 `"status": "warmup"` 时：
- 忽略 `pred_vol` 字段
- 继续接收 tick，等待预热完成
- 预热完成后开始使用 `pred_vol` 做策略

## 七、性能分析

### 7.1 延迟预算（20ms 内必须完成）

| 步骤 | 耗时 | 说明 |
|------|------|------|
| ZeroMQ 接收 | ~0.1ms | 本地 localhost |
| 特征计算 (Numba) | ~0.1ms | 增量算法 O(1) |
| 模型预测 | ~0.5ms | LightGBM 单样本 |
| 序列化 (orjson) | ~0.1ms | 极快 |
| ZeroMQ 发送 | ~0.1ms | 本地 localhost |
| **总计** | **~1ms** | 远低于 20ms |

### 7.2 结论

20ms 推送频率下，单次预测耗时约 **1ms**，有 **19ms 冗余时间**，完全满足实时性要求。

## 九、可能遇到的问题与解决方案

### 9.1 数据顺序问题

**问题**：tick 数据可能乱序到达。

**解决**：
- RealtimeFeatureComputer 使用 `timestamp` 而非接收时间作为时间戳
- 环形缓冲区按 timestamp 索引，不受乱序影响

### 9.2 模型文件损坏

**问题**：训练过程中断，生成不完整的模型文件。

**解决**：
- 训练完成后先写临时文件，验证无误后 rename
- 加载时检测文件完整性，损坏则跳过

### 9.3 预热期跨 UTC 0 点

**问题**：如果服务启动时间接近 UTC 0 点，预热期会跨越两天。

**解决**：
- 环形缓冲区存储的是相对数据（mid_price_tick 的变化），不依赖日期
- 预热期只关心 tick 数量，不关心日期
- 缓冲区满 360000 ticks 后自动开始正常预测

### 9.4 多币种并行

**问题**：需要同时服务多个 symbol。

**解决**：
- 每个 symbol 维护独立的 RealtimeFeatureComputer 实例
- 使用字典管理：`computers = {"XRPUSDT": ..., "DOGEUSDT": ...}`
- 模型文件按 symbol 目录分开存储
