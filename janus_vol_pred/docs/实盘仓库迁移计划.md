# `janus_prediction` 仓库规划方案（下班前保存版）

## Summary
目标是在 `/home/zhangzhanyi/workspace/janus_prediction` 中建立一个面向实盘部署的新仓库，支持四大功能：

1. 波动率每日定时训练
2. 波动率实时预测
3. 价格每日定时训练
4. 价格实时预测

当前阶段只正式落地波动率链路，价格链路先预留结构，后续由同事接入。

已锁定的设计决策：
- 环境管理：`venv + requirements`
- 仓库层级：按业务优先（`domain-first`）
- 配置方式：按“业务 + 运行角色”拆成 4 份配置文件
- 每日训练：`cron`
- 实时预测：前台 shell 脚本 + `tmux`
- 暂时不新建 `shared/`，避免提前抽象

## Key Changes
### 1. 仓库目录结构
建议新仓库采用如下结构：

```text
janus_prediction/
├── config/
├── requirements/
├── scripts/
├── janus_prediction/
│   ├── __init__.py
│   ├── volatility/
│   │   ├── daily_train/
│   │   └── realtime/
│   └── price/
│       ├── daily_train/
│       └── realtime/
└── README.md
```

说明：
- `scripts/`：给部署同事直接执行的脚本入口
- `volatility/`：当前阶段真正要迁移和跑通的代码
- `price/`：先建立清晰目录边界，暂不实现业务逻辑
- `shared/`：当前阶段先不建立，等真正出现稳定复用代码再抽

---

### 2. 配置文件策略
配置采用“按业务 + 角色拆分”的方式，不使用一个总配置文件。

建议配置模板文件：

```text
config/
  volatility.daily_train.example.yaml
  volatility.realtime.example.yaml
  price.daily_train.example.yaml
  price.realtime.example.yaml
```

实际本地部署配置：

```text
config/
  volatility.daily_train.yaml
  volatility.realtime.yaml
  price.daily_train.yaml
  price.realtime.yaml
```

规则：
- 仓库提交 `*.example.yaml`
- 真实运行配置 `*.yaml` 不提交，加入 `.gitignore`
- 训练机只维护 `volatility.daily_train.yaml`
- 实时机只维护 `volatility.realtime.yaml`
- 价格那两份配置先预留，不提前实现

这样做的原因：
- 波动率和价格由不同人开发，边界清晰
- 训练机和实时机路径、日志、结果目录不同
- 一台机器只关心自己那一份配置，部署最清楚

---

### 3. 环境与依赖
新仓库不再依赖 Poetry 作为部署前提，改为：

- 仓库提供 `requirements/`
- 部署机本地创建 `.venv`

建议依赖清单拆成：

```text
requirements/
  base.txt
  volatility_daily_train.txt
  volatility_realtime.txt
```

安装规则：
- 训练机安装：`base + volatility_daily_train`
- 实时机安装：`base + volatility_realtime`

提供一个统一环境安装脚本：

```text
scripts/bootstrap_venv.sh
```

它负责：
- 创建 `.venv`
- 升级 `pip`
- 按机器角色安装依赖

例如：
- `scripts/bootstrap_venv.sh volatility_daily_train`
- `scripts/bootstrap_venv.sh volatility_realtime`

依赖原则：
- 只保留部署机真正需要的依赖
- 不把 notebook、研究分析、回测类依赖搬进来
- 训练和实时各自只安装最小依赖集合

### 4. 波动率代码迁移范围
只迁移生产部署真正需要的代码，不整仓照搬。

波动率每日训练需要迁移的内容：
- 生产训练主流程
- 特征准备
- 原始数据依赖准备
- 模型发布
- 每日任务编排

波动率实时预测需要迁移的内容：
- 实时特征计算
- 模型加载
- 预测服务入口

迁移要求：
- 调整导入路径，使其适配新仓库结构
- 训练代码只读取 `volatility.daily_train.yaml`
- 实时代码只读取 `volatility.realtime.yaml`
- 保留“训练机产出 -> 同步到实时机”的设计
- 删除备份文件、研究态脚本、教程态代码、无关依赖

### 5. 启动脚本与部署体验
目标是让部署同事可以“一键做正确的事”。

训练机脚本：
- `scripts/bootstrap_venv.sh`
- `scripts/check_volatility_daily_train.sh`
- `scripts/run_volatility_daily_train.sh`
- `scripts/install_volatility_daily_cron.sh`

实时机脚本：
- `scripts/bootstrap_venv.sh`
- `scripts/check_volatility_realtime.sh`
- `scripts/run_volatility_realtime.sh`

约定：
- 每日训练：由 `cron` 调 `run_volatility_daily_train.sh`
- 实时预测：由同事在 `tmux` 中执行 `run_volatility_realtime.sh`
- 当前阶段不引入 `systemd`

## Interfaces / Runtime Behavior
### 公开使用入口
部署同事应只需要记住 shell 脚本，不需要记 Python 模块路径。

训练机入口：
- `scripts/bootstrap_venv.sh volatility_daily_train`
- `scripts/check_volatility_daily_train.sh`
- `scripts/run_volatility_daily_train.sh`
- `scripts/install_volatility_daily_cron.sh`

实时机入口：
- `scripts/bootstrap_venv.sh volatility_realtime`
- `scripts/check_volatility_realtime.sh`
- `scripts/run_volatility_realtime.sh`

---

### 配置读取规则
每个角色只读取自己那一份配置文件：

- 每日训练只读：
  - `config/volatility.daily_train.yaml`
- 实时预测只读：
  - `config/volatility.realtime.yaml`

当前阶段不做：
- 多配置继承
- base config + overlay
- 动态 merge
- 复杂 fallback

保持最直接清晰。
