# 每日定时重训前的数据补齐与特征准备方案（修正版）

## Summary
为 [janus_vol_pred](/home/zhangzhanyi/workspace/janus_vol_pred) 新增一条“训练前准备器”能力，不改动现有训练、特征计算、实时预测主链的职责边界。
每日 `UTC 00:00` 触发时，先根据当前训练配置动态计算所需历史特征天数，再检查这些日期的 `features_label` 是否齐全；若缺失，则补齐所需原始 `book_ticker`，生成缺失特征，最后才进入训练。若依赖最终仍不完整，则当天训练明确失败退出。

## Key Changes
### 1. 训练前准备范围动态取配置
- 所需特征日期数不写死。
- 动态使用：
  `required_feature_days = train_days + val_days`
- 当前配置下等于 `7 + 1 = 8`。
- 每次触发时，生成最近 `required_feature_days` 个目标特征日期。

### 2. `use_prev_day` 驱动原始数据依赖
- 特征准备逻辑不默认写死依赖前一天。
- 统一基于配置：
  `use_prev_day`
- 当 `use_prev_day = True` 时，生成目标日期 `D` 的 `features_label`，必须确保：
  - `book_ticker(D-1)`
  - `book_ticker(D)`
  都可读。
- 当 `use_prev_day = False` 时，只要求：
  - `book_ticker(D)`
- 这样和现有 [features_label.py](/home/zhangzhanyi/workspace/janus_vol_pred/janus_vol_pred/features_label.py) 的行为完全一致，避免准备层和计算层语义不一致。

### 3. 原始 `book_ticker` 的最优实践：解析逻辑解耦
采用解耦方案，不把“下载逻辑”塞进 `load_book_ticker()`。

职责划分：
- `load_book_ticker()` 只负责：读取一个标准化 `book_ticker` 文件。
- 新增“book_ticker 定位器”负责：按多来源顺序查找文件路径。
- 新增“训练前准备器”负责：在文件不存在时决定是否下载与转换。

推荐结构：

1. `resolve_book_ticker_path(symbol, date, ...)`
- 只负责查路径，不读数据，不下载。
- 查找顺序：
  - 同事维护的只读 Tardis 根目录
  - 自己历史补齐的本地根目录
- 返回命中的文件路径；若都没有，返回 `None`。

2. `load_book_ticker(...)`
- 内部调用 `resolve_book_ticker_path(...)`
- 拿到路径后继续沿用当前 CSV 读取、类型转换、时间戳换算逻辑
- 若未找到路径，抛 `FileNotFoundError`

3. `prepare_book_ticker_for_date(symbol, date, ...)`
- 上游准备函数
- 若 `resolve_book_ticker_path(...)` 已命中，则直接成功
- 若未命中，则尝试从公司行情库下载、转换并落盘
- 下载后再次校验能否被 `resolve_book_ticker_path(...)` 命中

这样做的好处：
- `features_label.py` 继续保持“读标准输入 -> 算特征”的单一职责
- 下载、调度、重试、失败退出都放在准备器层
- 未来若数据源再新增，不需要污染特征计算主链

### 4. 原始数据来源与查找顺序
对任意需要的 `book_ticker(date)`，查找顺序固定为：

1. 同事下载的只读 Tardis 数据
2. 自己历史从公司行情库补齐并保存的数据
3. 若仍不存在，则从公司行情库提取该日整天 `book_ticker`，转换成 Tardis 风格后保存到自己的目录

说明：
- 同事目录只读，不写入。
- 自己目录可写，专门保存 fallback 结果。
- 正常稳定运行后，通常只会缺“昨天”的数据，但逻辑允许补任意缺失日。

### 5. 原始数据保存规范
自己从公司行情库补齐的数据，保存为“按天目录 + Tardis 风格文件名”。

建议路径：
- `${output_root}/raw_book_ticker/{symbol}/book_ticker/{YYYY-MM-DD}/binance-futures_book_ticker_{YYYY-MM-DD}_{SYMBOL}.csv.zst`

示例：
- `/data/sigma/zzy/janus/results/vol_pred_prod/raw_book_ticker/XRPUSDT/book_ticker/2026-04-06/binance-futures_book_ticker_2026-04-06_XRPUSDT.csv.zst`

设计原则：
- 文件名与 Tardis 对齐，方便复用现有读取逻辑
- 自己的数据与同事目录彻底隔离
- 目录按天拆分，便于排查、重跑和清理

### 6. 行情库下载与格式转换
从旧项目 [get_database_data.py](/home/zhangzhanyi/workspace/hftbacktest_analysis/src/zzy/hftbacktest_analysis/hangqing_data_to_npz/helper/get_database_data.py) 与 [process_database_data.py](/home/zhangzhanyi/workspace/hftbacktest_analysis/src/zzy/hftbacktest_analysis/hangqing_data_to_npz/helper/process_database_data.py) 复用思路：

- 从公司 ClickHouse 拉取 `perpBestPrice`
- 使用已有 `convert_bbo_to_tardis_bbo()` 风格的转换逻辑，转成兼容当前项目的 `book_ticker` 表结构
- 输出列固定为：
  - `timestamp`
  - `local_timestamp`
  - `ask_amount`
  - `ask_price`
  - `bid_price`
  - `bid_amount`
- 保存为 `csv.zst`
- 保存前按时间排序
- 保存后必须做回读验证，确认当前 `load_book_ticker()` 能正常读入且行数大于 0

### 7. 每日定时重训的完整流程
1. 定时器在 `UTC 00:00` 触发
2. 读取配置，计算 `required_feature_days = train_days + val_days`
3. 生成最近 `required_feature_days` 个目标特征日期
4. 对每个目标日期 `D`：
   - 若 `features_label(D)` 已验证通过，则跳过
   - 否则根据 `use_prev_day` 计算所需原始数据日期集合
   - 对每个所需原始日期执行：
     - 查 Tardis 只读目录
     - 查自己历史补齐目录
     - 若都没有，则从公司行情库下载该日整天数据并转成标准格式落盘
   - 若原始数据准备完成，则生成 `features_label(D)`
   - 对生成结果做回读验证并写 `verified_records.csv`
5. 若所有目标特征日期都准备完成，则进入现有训练流程
6. 若任意关键日期特征仍缺失，则本次训练失败退出，不做脏训练

### 8. 配置新增建议
建议新增以下配置，不改已有训练配置含义：

```yaml
data_sources:
  tardis_book_ticker_root: "/path/to/read_only_tardis_root"
  fallback_book_ticker_root: "/data/sigma/zzy/janus/results/vol_pred_prod/raw_book_ticker"
  enable_hangqing_fallback: true

daily_retrain:
  enabled: true
  trigger_utc_hour: 0
  trigger_utc_minute: 0
  fail_if_feature_dependency_missing: true

hangqing:
  clickhouse_host: "13.231.173.109"
  clickhouse_port: 8123
  clickhouse_database: "marketData"
  clickhouse_user_env: "HANGQING_CH_USER"
  clickhouse_password_env: "HANGQING_CH_PASSWORD"
```

默认策略：
- `use_prev_day` 继续沿用现有配置
- Tardis 只读目录优先
- fallback 原始数据写入 `output_root` 下的新目录
- 若特征依赖最终不完整，则训练失败

## Test Plan
### 场景 1：最近所需天数的 `features_label` 全部已验证
- 预期：直接进入训练，不检查下载链路。

### 场景 2：目标日特征缺失，但 Tardis 原始数据齐全
- 预期：直接用 Tardis 原始数据生成特征，不访问公司行情库。

### 场景 3：目标日特征缺失，Tardis 无数据，但本地 fallback 原始数据已存在
- 预期：直接复用本地 fallback 原始数据，不重复下载。

### 场景 4：目标日特征缺失，且 `use_prev_day=True`
- 若 `D-1` 或 `D` 任一天原始数据缺失：
  - 预期：逐日补齐缺失原始数据后再生成 `features_label(D)`。

### 场景 5：正常调度下仅缺昨天原始数据
- 预期：仅下载昨天全天 `book_ticker`，转换后生成昨天对应特征。

### 场景 6：最近 8 天中缺的是更早某天
- 预期：仍会自动补齐那一天，不把逻辑限制为“只补昨天”。

### 场景 7：公司行情库下载或转换失败
- 预期：该原始日准备失败，相关特征无法生成，本次训练最终失败退出。

### 场景 8：生成出的 fallback `book_ticker` 文件不可读或为空
- 预期：删除坏文件或标记失败，不写验证记录，不继续基于坏数据生成特征。

## Assumptions
- 每日重训只需要准备 `train_days + val_days` 的特征日期。
- `test_days=1` 影响训练窗口切分，但不额外增加历史特征准备天数。
- 当前新增能力定位为“训练前准备器”，可以由 cron、systemd timer 或外部调度器触发，本方案不绑定具体调度器。
- 本次新增只覆盖 `book_ticker` fallback，不扩展到 trade/orderbook fallback。
- 公司行情库凭据应迁移到环境变量，不在新实现中继续硬编码。



# 每日定时重训前的数据补齐与特征准备方案

## Summary
为现有 [janus_vol_pred](/home/zhangzhanyi/workspace/janus_vol_pred) 新增一条“训练前自动补齐依赖”的准备链路，不改动已有特征计算、训练、实时预测主逻辑。
每日在 `UTC 00:00` 触发时，根据当前训练配置动态计算所需历史天数，检查最近所需天数的 `features_label` 是否齐全；若缺失，则逐日检查 `book_ticker` 原始数据，按“同事下载的 Tardis 数据 -> 自己历史落盘的数据 -> 公司行情库下载并转换”顺序补齐，随后生成缺失的 `features_label`，最后再进入训练。

## Key Changes
### 1. 调度与训练窗口
- 触发时间固定为 `UTC 00:00`，等价于北京时间 `08:00`。
- 所需特征天数不写死为 8，而是动态计算：
  `required_feature_days = train_days + val_days`
- 当前配置下：
  `7 + 1 = 8`
- 每次触发时，生成最近 `required_feature_days` 的日期列表并逐天检查。
- 默认目标是“为当天这次训练准备好完整依赖”；正常运行时通常只会缺昨天的数据，但逻辑上允许补任意缺失日。

### 2. 原始 `book_ticker` 的查找与补齐顺序
对每个缺失 `features_label` 的日期，按以下顺序检查 `book_ticker`：

1. 同事维护的 Tardis 标准数据目录
   说明：此目录只读，不在本功能内写入。
2. 自己历史补齐并落盘的本地 `book_ticker` 目录
   说明：这是本功能可写目录，保存从公司行情库补齐后的标准化文件。
3. 如果以上都不存在，则从公司行情库下载该日整天 `book_ticker` 数据，并转换为 Tardis 风格文件后落盘到自己的目录。

### 3. 原始数据落盘规范
采用“按天目录 + Tardis 风格文件名”的形式，便于复用与排查。

建议写入路径：
- 根目录：`${output_root}/raw_book_ticker`
- 结构：
  `${output_root}/raw_book_ticker/{symbol}/book_ticker/{YYYY-MM-DD}/binance-futures_book_ticker_{YYYY-MM-DD}_{SYMBOL}.csv.zst`

示例：
- `/data/sigma/zzy/janus/results/vol_pred_prod/raw_book_ticker/XRPUSDT/book_ticker/2026-04-06/binance-futures_book_ticker_2026-04-06_XRPUSDT.csv.zst`

设计原则：
- 同事的只读 Tardis 数据与自己补齐的数据分目录管理，避免权限冲突。
- 文件名保持与 Tardis 一致，方便后续统一加载逻辑复用。
- 目录按日拆分，方便人工核查、增量重跑与生命周期管理。

### 4. `book_ticker` 统一格式
从公司行情库取回的 `perpBestPrice` 数据，先转换成与当前 `load_book_ticker()` 兼容的 Tardis 风格 CSV：

目标列：
- `timestamp`
- `local_timestamp`
- `ask_amount`
- `ask_price`
- `bid_price`
- `bid_amount`

格式要求：
- 时间戳按当前项目约定保存为微秒级原始值，加载时仍由现有 `load_book_ticker()` 统一乘 `1000` 转为纳秒。
- 列名和类型与当前 `load_book_ticker()` 兼容，避免为 fallback 数据单独写一套特征生成逻辑。
- 保存格式优先 `csv.zst`。

### 5. `book_ticker` 解析逻辑调整
现有 `load_book_ticker()` 只查单层目录：
`Path(data_root) / symbol / "book_ticker"`

新逻辑改为“多来源解析 + 按天目录优先”：
- 优先查只读 Tardis 根目录：
  `{tardis_read_root}/{symbol}/book_ticker/{date}/{filename}`
- 再查本地补齐根目录：
  `{fallback_book_ticker_root}/{symbol}/book_ticker/{date}/{filename}`
- 若两处都没有，抛出“原始数据缺失”信号，由上游补齐流程决定是否下载。

### 6. 训练前准备总流程
每次调度的完整流程：

1. 读取配置，计算 `required_feature_days = train_days + val_days`
2. 生成最近 `required_feature_days` 的日期列表
3. 对每个日期检查 `features_label` 是否存在且通过验证
4. 若某日 `features_label` 缺失：
   - 检查同事的只读 Tardis `book_ticker`
   - 若无，再检查自己历史补齐的 `book_ticker`
   - 若仍无，从公司行情库拉取该日整天 `book_ticker`
   - 转成 Tardis 风格 `csv.zst` 后落盘
   - 基于统一格式生成该日 `features_label`
   - 回读验证并写入 `verified_records.csv`
5. 所有依赖日准备完成后，进入现有训练流程
6. 训练结果继续沿用现有验证、失败隔离、重试机制

### 7. 配置新增建议
建议在 YAML 中新增以下配置段，避免路径和策略硬编码：

```yaml
daily_retrain:
  enabled: true
  trigger_utc_hour: 0
  trigger_utc_minute: 0
  auto_prepare_features: true

data_sources:
  tardis_book_ticker_root: "/path/to/read_only_tardis_root"
  fallback_book_ticker_root: "/data/sigma/zzy/janus/results/vol_pred_prod/raw_book_ticker"
  prefer_tardis_first: true
  enable_hangqing_fallback: true

hangqing:
  clickhouse_host: "13.231.173.109"
  clickhouse_port: 8123
  clickhouse_database: "marketData"
  clickhouse_username_env: "HANGQING_CH_USER"
  clickhouse_password_env: "HANGQING_CH_PASSWORD"
```

默认策略：
- 只读 Tardis 目录优先
- 本地补齐目录次之
- 最后才访问公司行情库
- 数据库账号密码改为环境变量读取，不继续硬编码在业务脚本里

## Test Plan
### 场景 1：最近所需天数的 `features_label` 全部已验证
- 预期：直接跳过数据准备，进入训练。
- 不触发任何原始数据下载。

### 场景 2：缺少昨天的 `features_label`，但同事的 Tardis `book_ticker` 已存在
- 预期：直接使用只读 Tardis 数据生成缺失 `features_label`。
- 不访问公司行情库。

### 场景 3：缺少昨天的 `features_label`，Tardis 数据不存在，但本地历史补齐数据存在
- 预期：直接使用本地补齐 `book_ticker` 生成缺失 `features_label`。
- 不访问公司行情库。

### 场景 4：缺少某天 `features_label`，两类本地原始数据都不存在
- 预期：访问公司行情库拉该日整天数据，转换为 Tardis 风格并落盘，再生成 `features_label`。
- 成功后后续重跑应命中本地补齐文件，不重复下载。

### 场景 5：最近 8 天中缺的不是昨天，而是更早某天
- 预期：仍会识别并补齐那一天，确保训练依赖完整。
- 不将逻辑限制为“只能补昨天”。

### 场景 6：公司行情库下载成功但转换后文件不可读或行数为 0
- 预期：视为失败，不写验证记录，删除坏文件并按重试策略处理。

### 场景 7：单个日期补齐失败
- 预期：该日期标记失败并记录日志。
- 本次训练不应在依赖不完整时盲目启动；应整体判定“训练前准备失败”。

## Assumptions
- 每日重训只需要准备 `train_days + val_days` 的特征数据，不额外要求 `test_days` 对应的未来特征。
- 生成 `features_label` 时所需的“前一天原始数据补充”仍由现有 `use_prev_day=True` 机制处理，因此若某个目标日需要前一天原始数据，上游准备流程也要确保该前置原始日可读。
- 同事维护的 Tardis 数据目录只读，不写入。
- 自己补齐的原始 `book_ticker` 数据统一写入 `${output_root}/raw_book_ticker/...`。
- 行情库补齐只针对 `book_ticker`，不扩展到 trades/orderbook，除非后续训练链路明确需要。
- 本次新增功能定位为“训练前准备器”，可由 cron、systemd timer 或外层调度平台触发；本方案不强制绑定具体调度器。
