## 一、整体数据流

```
Go 程序（交易所行情）
    │
    │  ZeroMQ PUSH（不定频，每条 bookTicker 立即推送）
    ▼
predict_service.py（Python 预测服务）
    │
    ├─ 过滤：只处理 event=bookTicker，按 symbol 分流
    │
    ├─ 维护每个 symbol 的"最新 tick 缓存"（last_tick）
    │
    ├─ 定时器：每 20ms 触发一次采样
    │       └─ 取 last_tick → 写入环形缓冲区
    │
    ├─ RealtimeFeatureComputer（Numba 增量计算）
    │       ├─ 预热期（< 360000 ticks）→ status: warmup
    │       └─ 正常期 → 计算 vol_1m ~ vol_120m + 时间特征
    │
    ├─ 模型推理（LightGBM + QuantileTransformer）
    │       └─ 模型查找：当天 → 前一天 → 依次往前
    │
    └─ ZeroMQ PUSH → Go 程序（每 20ms 一次）
```

## 二、数据格式对齐

### 2.1 实盘接收的 bookTicker 格式

```json
{
  "event": "bookTicker",
  "symbol": "XRPUSDT",
  "updateID": 10240743962314,
  "bidPrice": 1.31,
  "bidQty": 7274,
  "askPrice": 1.3101,
  "askQty": 34277.2,
  "tradeTime": 1775120538048000,
  "updateTime": 1775120538049782
}
```

### 2.2 字段映射到历史数据格式

| 实盘字段 | 历史字段 | 说明 |
|---------|---------|------|
| `bidPrice` | `bid_price` | 买一价 |
| `askPrice` | `ask_price` | 卖一价 |
| `bidQty` | `bid_amount` | 买一量 |
| `askQty` | `ask_amount` | 卖一量 |
| `tradeTime × 1000` | `timestamp` (ns) | 交易所时间，微秒→纳秒 |
| `updateTime × 1000` | `local_timestamp` (ns) | 本地收到时间，微秒→纳秒 |

### 2.3 时间戳选择：用 `tradeTime`

- `tradeTime` = 交易所撮合时间，与历史数据的 `timestamp` 语义一致
- `updateTime` = 本地处理时间，有网络延迟，不用于特征计算
- 转换：`timestamp_ns = tradeTime * 1000`

## 三、20ms 采样机制

### 3.1 设计思路

实盘推送是**不定频**的（交易所有更新就推），我们需要**规整到 20ms 网格**。

方案：**"最新 tick 缓存 + 定时采样"**

```
实盘推送（不定频）：
  t=0ms:   bookTicker A
  t=3ms:   bookTicker B
  t=11ms:  bookTicker C
  t=18ms:  bookTicker D
  t=25ms:  bookTicker E

定时采样（每 20ms）：
  t=20ms:  取 last_tick = D（20ms 内最新的一条）
  t=40ms:  取 last_tick = E（40ms 内最新的一条）
```

这与历史数据的 `asof_join backward` 语义完全一致：**取 < T 时刻的最新一条**。

### 3.2 实现方式

```python
# 每个 symbol 维护一个 last_tick
last_tick: dict[str, dict] = {}  # symbol -> 最新 bookTicker

# ZeroMQ 接收线程：收到就更新 last_tick
def recv_loop():
    while True:
        msg = orjson.loads(receiver.recv())
        if msg.get("event") == "bookTicker":
            symbol = msg["symbol"]
            last_tick[symbol] = msg

# 定时采样线程：每 20ms 触发
def sample_loop():
    while True:
        t_start = time.perf_counter()
        now_ns = time.time_ns()
        
        for symbol, computer in computers.items():
            tick = last_tick.get(symbol)
            if tick is None:
                continue
            
            bid = tick["bidPrice"]
            ask = tick["askPrice"]
            ts_ns = tick["tradeTime"] * 1000  # 微秒 → 纳秒
            
            result = computer.update_and_predict(bid, ask, ts_ns, now_ns)
            sender.send(orjson.dumps(result))
        
        # 精确 20ms 间隔
        elapsed = time.perf_counter() - t_start
        sleep_time = 0.020 - elapsed
        if sleep_time > 0:
            time.sleep(sleep_time)
```

## 四、RealtimeFeatureComputer（Numba 实现）

### 4.1 为什么用 Numba

- 每 20ms 调用一次，要求 < 1ms 完成
- Polars 批处理太重（每次重建 DataFrame）
- Numba JIT 编译后接近 C 速度，增量计算 O(1)

### 4.2 环形缓冲区设计

```python
MAX_TICKS = 360_000  # 120min × 60s × 1000ms/20ms = 360000

# 存储 mid_price_tick 的变化量（mid_chg）
buffer = np.zeros(MAX_TICKS, dtype=np.float64)
buffer_head = 0   # 下一个写入位置
buffer_count = 0  # 已写入数量
```

### 4.3 增量波动率计算

```python
@njit
def compute_rolling_std(buffer, head, count, window):
    """从环形缓冲区计算最近 window 个值的标准差"""
    if count < window:
        return np.nan
    
    s = 0.0
    s2 = 0.0
    for i in range(window):
        idx = (head - 1 - i) % MAX_TICKS
        v = buffer[idx]
        s += v
        s2 += v * v
    
    mean = s / window
    var = s2 / window - mean * mean
    return np.sqrt(var) if var > 0 else 0.0
```

### 4.4 预热期判断

```python
WARMUP_TICKS = 360_000  # 120 分钟

if buffer_count < WARMUP_TICKS:
    # 预热期，不输出预测
    return {"symbol": symbol, "timestamp": now_ns, "status": "warmup"}
```

### 4.5 与历史批处理的一致性

| 特征 | 历史（Polars） | 实盘（Numba） | 一致性 |
|------|--------------|-------------|--------|
| `mid_chg` | `mid_tick.diff()` | `mid_tick[t] - mid_tick[t-1]` | ✅ |
| `vol_Nm` | `rolling_std(window=N*60*50)` | 环形缓冲区增量 std | ✅ 需验证 |
| `vol_ratio` | 条件判断 | 同逻辑 | ✅ |
| 时间特征 | sin/cos 公式 | 同公式 | ✅ |

### 5.2 查找逻辑

```python
def find_latest_model(results_dir: Path, symbol: str, max_lookback: int = 7) -> Path | None:
    """
    从今天开始往前找，最多找 max_lookback 天
    优先当天，没有则找前一天，依次往前
    """
    today = datetime.utcnow().date()
    for i in range(max_lookback):
        date = (today - timedelta(days=i)).strftime("%Y-%m-%d")
        model_dir = results_dir / date / symbol
        if (model_dir / "model.txt").exists():
            return model_dir
    return None
```

### 5.3 模型热更新

每次采样时检查 `model.txt` 的 mtime，变化则重新加载：

```python
def check_and_reload(self):
    model_dir = find_latest_model(results_dir, self.symbol)
    if model_dir is None:
        return
    
    current_mtime = (model_dir / "model.txt").stat().st_mtime
    if current_mtime != self.model_mtime:
        try:
            self.model = lgb.Booster(model_file=str(model_dir / "model.txt"))
            self.qt = joblib.load(model_dir / "quantile_transformer.pkl")
            with open(model_dir / "feature_cols.json") as f:
                self.feature_cols = json.load(f)
            self.model_mtime = current_mtime
            logger.info(f"[{self.symbol}] 模型已更新: {model_dir}")
        except Exception as e:
            logger.error(f"[{self.symbol}] 模型加载失败，保留旧模型: {e}")
```

## 十、注意事项

1. **`tradeTime` 可能不单调**：实盘数据偶尔乱序，环形缓冲区按接收顺序写入，不按时间戳排序（与历史 asof_join 有细微差异，需验证影响）

2. **`last_tick` 线程安全**：接收线程和采样线程共享 `last_tick`，Python dict 的单次赋值是原子的，不需要加锁

3. **首次启动无模型**：服务启动时如果找不到任何模型，输出 `status: no_model`，等模型上传后自动热更新

4. **多 symbol 支持**：每个 symbol 独立的 `RealtimeFeatureComputer` 实例和 `ModelLoader` 实例，互不干扰



## tick_size 动态更新

### 设计思路

`tick_size` 由同事每分钟推送一次，服务端维护 per-symbol 内存缓存。
接收线程收到即更新，采样线程每 20ms 直接读缓存，无需主动轮询。

### 实际消息格式

```json
{
  "Event": "symbol",
  "Symbol": "XRPUSDT",
  "TickSize": 0.0001,
  "StepSize": 0.1,
  "PricePrecision": 4,
  "QtyPrecision": 1,
  "MinQty": 0.1,
  "MinNotional": 5
}
注意：字段名首字母大写（Event、Symbol、TickSize）。

处理逻辑

tick_size_cache: dict[str, float] = {}

# 接收线程：收到就更新
if msg.get("Event") == "symbol":
    tick_size_cache[msg["Symbol"]] = float(msg["TickSize"])

# 采样线程：每 20ms 直接读，还没收到则跳过
tick_size = tick_size_cache.get(symbol)
if tick_size is None:
    continue  # 等待 symbol 消息到达
注意事项
tick_size_cache 单次赋值是原子的，两个线程共享无需加锁
服务启动后等待第一条 symbol 消息到达才开始该 symbol 的采样
同事需在预测服务启动前先启动推送服务

## 实时 tick 缓存与采样

### 缓存策略

只保留 per-symbol 最新一条 bookTicker，用 `updateID` 判断新旧：

​```python
latest_tick: dict[str, dict] = {}  # symbol -> 最新一条 bookTicker

# 接收线程：收到 bookTicker 就更新缓存
if msg.get("event") == "bookTicker":
    symbol = msg["symbol"]
    cached = latest_tick.get(symbol)
    if cached is None or msg["updateID"] > cached["updateID"]:
        latest_tick[symbol] = msg
    # updateID 更小则丢弃（乱序/重复消息）
​```

### 定时采样（每 20ms）

​```python
# 采样线程：每 20ms 读一次缓存
tick = latest_tick.get(symbol)
if tick is None:
    continue  # 还没收到该 symbol 的数据

timestamp_ns = tick["tradeTime"] * 1000  # 微秒 → 纳秒
bid_price = float(tick["bidPrice"])
ask_price = float(tick["askPrice"])
​```

### 时间戳说明

| 字段 | 对应历史字段 | 含义 |
|------|------------|------|
| `tradeTime` | `timestamp` | 交易所撮合时间（微秒），转纳秒用于特征计算 |
| `updateTime` | `local_timestamp` | 本地接收时间，不参与特征计算 |
| `updateID` | — | 单调递增 ID，用于判断消息新旧，丢弃乱序消息 |

# 编程习惯

1.尽可能重要参数（包括这次跑什么币种，时间）和各项路径，就都在一个yaml文件中进行管理。
2.尽可能在各个关键环节，用loguru去打印日志。
3.尽可能有增量更新的功能，防止跑到一半断掉需要全部再跑。
4.如果需要用多进程进行加速，那么就用多进程，如果用不了，那么就算了。
5.可以打印程序进度，这样方便监控。
6.如无必要，不要写太冗余的代码，有第一定的防御机制即可，代码注重运行高效，此外也希望代码精简，不要太冗余。
7。在最终的多symbol多date的运行中，如果一个币一天的有问题，别影响其他币其他日，跳过即可。
8. 数据生成后需要验证：生成 最终数据如parquet或者npz 后重新读取验证（能打开且行数>0），通过后记录到 verified_records.csv，若不通过则删去重新生成（最多重试3次）。增量检查不仅基于文件是否存在，还要基于通过验证的记录中是否包括该文件：
不在验证记录中 → 不急着重新生成，先试着验证一下已有文件。验证通过 → 补录到记录，跳过。验证失败 → 再删除重生
